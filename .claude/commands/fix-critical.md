# fix-critical

Найди топ-5 критических проблем в бэкенде на текущий момент и исправь их.

## Порядок выполнения

### 1. Исследование кодовой базы

Проведи анализ по следующим категориям (от наиболее к наименее критичному).
Для каждой категории ищи **конкретные файлы и строки**, а не общие формулировки.

#### Категория A — Изоляция workspace (КРИТИЧНО)

Найди все места, где данные одного workspace могут утечь в другой:

- `findAll()` / `findAllBy...()` без фильтра `workspaceId` в сервисах и контроллерах, которые работают в контексте конкретного workspace
- Вызовы `WorkspaceContext.get()` без предварительной проверки `WorkspaceContext.isSet()` в путях кода, где workspace может быть не установлен
- Методы репозиториев без `workspaceId`-параметра, которые вызываются из workspace-aware слоя

#### Категория B — N+1 запросы (ВЫСОКАЯ КРИТИЧНОСТЬ)

- Циклы, внутри которых вызывается `repository.findById(...)`, `repository.findByX(singleId)` или любой другой запрос на каждую итерацию
- Цепочки `stream().map(id -> repo.findById(id))` без предварительной batch-загрузки
- Загрузка связей через `@OneToMany` / `@ManyToMany` без `JOIN FETCH` или batch-запроса

#### Категория C — Надёжность (СРЕДНЯЯ КРИТИЧНОСТЬ)

- `Integer.parseInt()`, `Long.parseLong()` без try-catch на данных из внешних источников (HTTP-заголовки, API-ответы, пользовательский ввод)
- Итерации по внешним коллекциям (API-ответы, DB-результаты) без защиты от partial failure (один сломанный элемент валит весь пакет)
- Scheduler-методы без try-catch вокруг обработки каждого элемента

#### Категория D — Корректность метрик и ключей (СРЕДНЯЯ КРИТИЧНОСТЬ)

- Хардкоженные строковые ключи метрик вместо `Metric.XXX.key()`
- Несоответствие ключей между Java (где метрика пишется) и Thymeleaf-шаблоном (где читается)
- Использование camelCase строк там, где в `metricsJson` хранятся snake_case ключи

#### Категория E — Качество API (НИЗКАЯ КРИТИЧНОСТЬ)

- `findAll()` в REST-контроллерах без пагинации (риск OOM при большом числе записей)
- Отсутствие `@Transactional(readOnly = true)` на методах, которые только читают данные через несколько репозиториев

### 2. Ранжирование

После анализа выбери **ровно 5 проблем**, отсортированных по критичности (1 = самая критичная).
Для каждой укажи:

- Файл и номер строки
- Категорию (A/B/C/D/E)
- Одно предложение — почему это проблема

Представь список пользователю и получи подтверждение («исправляем» или аналогичное) перед тем, как приступить к исправлениям.

### 3. Исправление

Исправляй проблемы **по одной**, в порядке убывания критичности:

1. Прочитай файл целиком перед редактированием
2. Внеси минимальное изменение, решающее именно эту проблему — не рефакторь соседний код
3. После каждого исправления выполни:
   ```bash
   ./gradlew build -x test 2>&1 | tail -15
   ```
   Убедись, что build SUCCESSFUL. Если нет — исправь ошибку компиляции / static analysis перед переходом к следующей проблеме.

### 4. Финальная проверка

После всех пяти исправлений:

```bash
./gradlew build 2>&1 | tail -20
```

Если BUILD SUCCESSFUL — создай один коммит со всеми изменениями:

```
refactor: fix top-N critical issues found by /fix-critical

- #1 <краткое описание>
- #2 <краткое описание>
...

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

## Правила

- **Только интеграционные тесты** — не создавай и не изменяй unit-тесты
- **Минимальные изменения** — не трогай код, который не относится к исправляемой проблеме
- **Не добавляй комментарии** к коду, который не менял
- **Не добавляй обработку ошибок** для сценариев, которые в данном коде невозможны
- При добавлении импортов соблюдай порядок: `THIRD_PARTY_PACKAGE ### STANDARD_JAVA_PACKAGE ### STATIC` (Checkstyle)
- Используй `@SuppressWarnings("checkstyle:IllegalCatch")` при необходимости поймать `Exception`
- `WorkspaceContext.get()` **никогда** не возвращает null — он бросает исключение; не проверяй результат на null
- Перед вызовом `WorkspaceContext.get()` в путях, где workspace может быть не установлен, проверяй `WorkspaceContext.isSet()`
